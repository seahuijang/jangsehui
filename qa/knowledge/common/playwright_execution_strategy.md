# Playwright UI 자동화 실행 전략

## 기본 원칙
- 브라우저 제어는 반드시 Playwright MCP를 사용한다
- bash로 Playwright를 직접 실행하지 않는다
- 로그아웃은 반드시 UI 흐름으로 처리한다 (쿠키 삭제·JS 강제 로그아웃 금지)

---

## 뷰포트 설정 (테스트 시작 전 필수)
```
PC 웹:     width: 1440, height: 900
모바일 웹:  width: 430,  height: 932
```
반드시 테스트 시작 전 뷰포트를 설정한다.

---

## 실행 단계 (Phase)

### Phase 0 — 환경 준비 (모든 TC 실행 전 필수)
1. 관리자 계정으로 LACMS 로그인
2. 사전 구성된 테스트 상품 URL 접근 가능 여부 확인
3. 상품 접근 불가 시 LACMS에서 신규 생성
4. 테스트에 필요한 상품 번호·이름 메모

### Phase 1 — 장바구니 세팅
각 TC 실행 전 사전 조건에 맞게 장바구니를 구성한다:
1. 컬리몰에서 올바른 계정으로 먼저 로그인
2. 기존 장바구니 상태 확인
3. 필요한 상품 검색 후 장바구니 담기
4. 수량·선택 상태를 사전 조건에 맞게 조정
5. 이후 TC의 테스트 단계 실행

### Phase 2 — 컬리나우 테스트 (로그인 없는 환경)
컬리나우는 자체 로그인 기능이 없으므로 아래 순서를 반드시 따른다:
1. 컬리몰에서 필요한 계정으로 먼저 로그인
2. 세션 유지 상태에서 컬리나우 URL로 이동
3. 배송지가 올바른지 확인 후 필요 시 변경
4. 컬리나우에서 장바구니·주문서 테스트 진행
- 계정 전환 시: 컬리몰 로그아웃 → 다른 계정 로그인 → 컬리나우 재접속

### Phase 3 — 주문·취소 플로우
1. 장바구니에서 "주문하기" 버튼 클릭 → 주문서 진입
2. 주문서 상단 멤버스 적립금 문구 확인
3. "적립금·컬리캐시" 섹션 → "모두사용" 버튼 클릭 → 적립금 전액 적용
4. "결제하기" 버튼 클릭하여 주문 완료
5. 주문 후: 마이페이지 > 주문내역 → 주문 취소
6. 취소 후: 장바구니 재진입하여 적립금 복원 확인

### Phase 4 — 계정 전환 테스트
멤버스 ↔ 비멤버스 전환 TC는 반드시 UI 로그아웃 흐름 사용:
1. 마이페이지(또는 햄버거 메뉴)에서 "로그아웃" 버튼 클릭
2. 로그아웃 완료 확인 후 다른 계정으로 로그인
3. 장바구니 진입하여 문구 표시 여부 확인
4. 컬리나우 테스트 필요 시 → 로그인 후 컬리나우 URL로 이동

### Phase 5 — 특수 상품·쿠폰 처리
1. 사전 구성된 상품 URL 우선 사용
2. 추가 상품 필요 시 컬리몰에서 키워드 검색:
   - 100원딜 → "100원" 검색
   - 럭키박스 → "럭키박스" 검색
   - 컬리카드 → "컬리카드" 검색
3. 쿠폰 필요 시 LACMS에서 생성 후 테스트 계정에 발급
4. 적합한 상품 없으면 LACMS에서 신규 상품 생성

## BLOCKED 방지 규칙

| 상황 | 해결 방법 |
|------|---------|
| 사전조건 "장바구니에 상품 담긴 상태" | 직접 검색하여 담기 |
| 특정 적립금 잔액 필요 | 잔액별 계정 매핑 참고 |
| 적립금 사용 불가 상품 필요 | 사전 구성 URL 사용 또는 키워드 검색 |
| 쿠폰 필요 | LACMS에서 생성·발급 |
| 상품 없음 | LACMS에서 신규 생성 |
| 주문 필요 | 적립금 "모두사용" → "결제하기" 클릭 |
| 취소 필요 | 마이페이지에서 실제 취소 |

### 자동 BLOCKED 처리 (시도하지 않고 즉시 BLOCKED 처리)
- 사전조건에 "30만원 이상 구매 적립금 보유" 또는 "263번 적립 유형" 명시된 경우
- 사유: "다음달 정기결제 배치 작업이 필요한 케이스로, 수동 테스트 대상 (AI 테스트 제외)"

## 상품 검색 전략
1. 컬리몰/컬리나우 메인 검색창 사용
2. 일반 상품: "우유", "계란", "두부" 등 키워드 검색
3. 저가 상품: 가격 필터로 1,000~2,000원 범위 검색
4. 적립금 제한 상품: "럭키박스", "100원" 등 키워드 검색
5. 검색 결과 없으면 카테고리 탐색
6. 상품 가격은 TC와 정확히 일치하지 않아도 됨 (±500원 범위 허용)

---

## 결과 판정 기준
- **PASS**: 기대 결과와 실제 결과가 일치
- **FAIL**: 기대 결과와 실제 결과가 불일치 → 계속 다음 TC 진행 (중단 금지)
- **BLOCKED**: 위 모든 방법 시도 후에도 실행 불가 시 → 상세 사유 기록

---

## 결과 보고서 출력 형식

### 전체 결과 (summary.csv)
```
No,Priority,Type,TC Title,Pre-Condition,Test Step,Expected Result,Actual Result,Result,Notes,Screenshot
```

### 실패 상세 (fail-detail.csv)
```
No,Priority,TC Title,Expected Result,Actual Result,Fail Reason,Screenshot
```

### BLOCKED 상세 (blocked-detail.csv)
```
No,Priority,TC Title,Pre-Condition,Blocked Reason,Suggested Action
```

CSV 규칙:
- UTF-8 BOM 인코딩 (한글 Excel 호환)
- 쉼표 포함 값은 쌍따옴표로 감싸기
- 헤더 행 포함
- 스크린샷: test-results/TC-{No}/ 폴더에 저장

---

## 테스트 완료 후 — 도메인 지식 문서 생성
모든 테스트 완료 후 아래 내용을 knowledge 폴더에 문서로 저장한다:
- 주요 정책 (계산 공식, 적립금 조건, 사용자 유형별 규칙)
- 테스트 주요 포인트 (자주 FAIL되는 TC, 엣지 케이스)
- 화면 플로우 (주요 사용자 흐름, 상태 변화)
- 다음 테스트 참고사항 (사전 준비 데이터, 추가 TC 제안)
- 기존 파일 있으면 덮어쓰지 않고 내용 추가(Append)

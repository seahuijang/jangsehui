
 QA 기초 개념

## QA란 무엇인가
QA(Quality Assurance)는 시스템이 비즈니스 요구사항과 정책을 올바르게 구현했는지 검증하는 활동입니다.
발주 담당자 관점에서 QA는 **"계약한 기능이 약속대로 동작하는가"** 를 확인하는 것입니다.

---

## 테스트 레벨

| 레벨 | 설명 | 발주 담당자 관여도 |
|------|------|-----------------|
| 단위 테스트 | 개발자가 코드 단위로 검증 | 낮음 |
| 통합 테스트 | 여러 기능이 연결되어 동작하는지 검증 | 중간 |
| 시스템 테스트 | 전체 시스템이 요구사항대로 동작하는지 검증 | 높음 |
| 인수 테스트(UAT) | 발주 담당자가 직접 요구사항 충족 여부 확인 | **핵심** |

---

## 발주 담당자가 집중해야 할 테스트 관점

### 1. 기능 적합성
- 요구사항 명세서에 정의된 기능이 모두 구현되었는가? 
- 비즈니스 규칙이 코드에 정확히 반영되었는가?
- 불명확하 요구사항은 추측하지 말고 질문 

### 2. 정책 준수
- 제한 조건(수량 한도, 기간 제한 등)이 올바르게 적용되는가?
- 예외 상황에서 시스템이 정책대로 차단/처리하는가?

### 3. 데이터 정합성
- 처리 결과가 관련된 모든 화면과 시스템에 정확히 반영되는가?
- 수량·금액·상태 값이 일관되게 유지되는가?

### 4. 사용자 경험
- 오류 발생 시 사용자가 이해할 수 있는 안내 메시지가 표시되는가?
- 처리 흐름이 업무 절차와 일치하는가?

---

## 좋은 TC의 조건
1. **명확성**: TC 제목만 봐도 무엇을 테스트하는지 알 수 있다
2. **추적 가능성**: 어떤 요구사항/정책을 검증하는지 근거가 있다
3. **재현 가능성**: 누가 실행해도 동일한 결과가 나온다
4. **독립성**: 다른 TC에 의존하지 않고 단독 실행 가능하다
5. **간결성**: 꼭 필요한 단계만 포함한다
6. **정확성**: 불명확한 요구사항은 질문.md에 정리. 추측 금지.
7. **실행 가능성**: 누구나 동일하게 실행할 수 있도록 구체적인 값 명시.
8. **완전한 커버리지**: Happy Path + Edge Case 모두 포함.
9. **파일 분리**: UI/UX 테스트케이스와 보안 테스트케이스는 별도 CSV로 분리

## ⚠️ 필수 커버리지 체크 (반드시 포함)

### 1. 핵심 비즈니스 기능 (UI/UX 테스트케이스에 포함)

**반드시 테스트 케이스에 포함해야 할 기능:**
입력 문서에서 아래 영역이 해당되는 경우, 반드시 TC에 포함합니다.

| 영역 | 반드시 포함할 케이스 |
|------|----------------------|
| 로그인 | 정상 로그인, 아이디/비밀번호 오류, 빈값, 세션 만료, 로그아웃 후 뒤로가기 |
| 메뉴/네비게이션 | 권한별 메뉴 노출, 미권한 URL 직접 접근, 탭 전환, 브라우저 뒤로가기 |
| 검색 | 키워드 단/복합, 검색 결과 없음, 필터 조합(AND), 전체 초기화, 날짜 범위 오류 |
| 리스트/테이블 | 데이터 로딩, 빈 목록, 정렬, 페이지네이션(첫/마지막), 필터 후 1페이지 복귀 |
| 버튼 | 중복 클릭 방지, 비활성 조건, API 성공/실패 처리 |
| 다운로드 | 파일 형식/인코딩, 필터 반영 여부, 전체 데이터 포함 |
| 상세 화면 | 데이터 정합성, 수정 저장/취소, 삭제 확인 팝업, 연관 데이터 삭제 제한 |
| 화면 UI | 필수값 표시, 유효성 오류 위치, 토스트 메시지, 빈 상태 화면, 오류 화면(403/404/500) |


--
### 2. 우선순위 기준
- 🔴 Critical — 서비스 핵심 기능, 반드시 통과
- 🟠 High — 주요 UX 영향, 출시 전 검증 필요
- 🟡 Medium — 일반 기능, 이슈 발생 시 대응 가능
- 🟢 Low — 접근성·마이너 UX, 여유 시 검증

--
## 출력 규칙

1. 공통 영역 정책을 기준으로 해당되는 영역을 먼저 판단합니다.
2. 입력 문서에서 도메인 특화 케이스를 추가로 추출합니다.
3. 중복 케이스는 병합하고, 누락 케이스는 공통 정책 기준으로 보완합니다.
4. 출력은 **복붙 가능한 마크다운 테이블** 형태로만 작성합니다.
5. 설명, 전문, 요약 없이 **테이블만** 출력합니다.
6. 테스트 케이스 형식은 tc_writing 문서 기반으로 작성합니다.

# Part 1. GitHub 레포지토리 분석 (Git 기반 입력인 경우 수행)

> PRD·정책서만 제공된 경우 이 파트를 건너뛰고 Part 2부터 시작합니다.

## Step 1. 레포 루트 파악

레포에 진입하면 다음 파일을 **이 순서대로** 먼저 읽습니다.

**① `README.md`**
아래 항목이 있으면 정책서 뼈대로 활용합니다.
- 서비스 목적 및 대상 시스템
- 모듈 구성 및 의존 관계
- 외부 서비스 연동 목록
- 데이터 저장소
- CI/CD 구조 및 브랜치 전략

**② `build.gradle` / `settings.gradle` (또는 `pom.xml`)**
- 멀티모듈 구조 확인
- 주요 라이브러리·프레임워크 버전 확인
- 모듈 간 의존 관계 확인

**③ `CLAUDE.md` / `CONTRIBUTING.md`**
- 작업 프로세스 및 브랜치 컨벤션
- 배포 명령어 패턴 및 팀 운영 규칙

---

## Step 2. 모듈별 코드 분석

우선순위: **Controller → Service → 외부 연동 Client → Config**

| 대상 | 경로 패턴 | 확인 항목 |
|------|-----------|-----------|
| Controller | `**/*Controller.java` | 엔드포인트 URL, HTTP 메서드, 요청/응답 파라미터, 인증 여부 (`@AuthenticationPrincipal`, `@PreAuthorize`) |
| Service | `**/*Service.java` | 핵심 비즈니스 로직 흐름, 트랜잭션 경계 (`@Transactional`), 이벤트 발행 지점 |
| Feign/RestClient | `**/client/`, `**/infra/` | 호출 대상 서비스명, 주요 API 메서드, 호출 조건 |
| application.yml | `src/main/resources/` | DB/Redis 연결 설정, 외부 서비스 URL, 환경별 설정 분리 여부 |

---

## Step 3. 인프라 및 배포 구성 분석

- **`.github/workflows/`** — CI/CD 파이프라인 단계별 역할, 환경별(dev/stg/perf/prod) 배포 트리거 조건
- **`deploy/` / `ecs-deploy/`** — 컨테이너 구성, 환경변수, 환경별 차이점

# Part 2. QA 테스트 가이드 작성

## Step 1. 도메인 구조 파악

문서 또는 분석 결과에서 아래 항목을 추출합니다.

- 핵심 객체(Entity)와 상태값(Status Enum) 목록
- 상태 전이 트리거 (유저 액션 / 배치 / 자동화)
- 외부 시스템 연동 지점 (API, RMS, 알림 등)
- 정책상 분기 조건 (OR / AND 복합 조건 포함)

---

## Step 2. 상태 전환 매트릭스 작성

| 현재 상태 | 트리거(Action) | 전이 상태 | 기대 결과 | 체크포인트 |
|-----------|----------------|-----------|-----------|------------|
| `STATE_A` | 유저액션/배치 | `STATE_B` | DB+UI 반영 | 중복처리/롤백 |

- 정상 전이 외 **불가 전이(Invalid Transition)** 케이스도 명시
- 자동화(배치/타이머) 기반 전이는 트리거 조건과 시점을 명확히 기재

---

## Step 3. 기능별 검증 항목

각 기능/메뉴 단위로 표 형식으로 작성합니다. (항목 | 검증 내용 | 기대 결과)

1. **노출 조건** — 어떤 상태/권한/데이터 조건일 때 노출되는지
2. **데이터 정합성** — UI 표시값 vs DB 적재값, 수량/날짜/상태값 일치 여부
3. **분기 로직** — OR/AND 복합 조건은 케이스별로 분리
4. **알림/연동** — 발송 시점, 수신 대상, 변수 치환 정확성
5. **동시성/중복** — 더블클릭, 중복 요청, Race Condition 가능성
6. **예외 경로** — 네트워크 오류, 권한 없음, Empty 상태, 최대치 데이터

---

## Step 4. Git 변경분 기반 영향도 분석

> Git Diff / PR이 제공된 경우 작성합니다. Part 1 분석 결과와 연계하여 작성합니다.

- **Side-Effect 범위** — 수정된 공통 모듈/함수가 영향을 주는 다른 기능 목록
- **API 변경** — Request/Response 파라미터 변경 시 연관 화면 전수 확인
- **DB Schema 변경** — 기존 데이터(Legacy) 호환성 및 마이그레이션 검증
- **환경별 설정** — 운영/스테이징 환경 변수 분기 정확성

---

## Step 5. 최종 산출물

**① ASCII 상태 흐름도** (상태 전이가 있는 경우)

**② QA 우선순위 체크리스트**
- Blocker/Critical — 주요 기능 작동 불가, 데이터 손실
- Major — 기능은 동작하나 기획과 다름
- Minor — UI 오정렬, 오타 등

**③ ⚠️ 정책 확인 필요 항목** — 문서에서 모호하거나 누락된 예외 케이스

# 작성 규칙

- 명시된 정책은 기대 결과를 수치/상태값으로 구체화
- 모호하거나 코드·문서에서 확인 불가한 항목은 **추정하지 않고 ⚠️ 로 분리**
- 개발 구현이 아닌 **운영자/공급사 시나리오 기준**으로 작성
- 알림 메시지가 있으면 변수 단위로 치환 검증 항목 추가
- Git 분석 시 확인 불가한 항목은 "확인 불가" 또는 "코드에 명시되지 않음"으로 명시

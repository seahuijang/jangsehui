## 0. 사용자 유형 정의

| 유형 | 정의 | 주요 권한 |
|------|------|----------|
| **파트너사용자(공급사)** | 컬리 프로모션에 참여 신청하는 공급사 담당자 | 본인 공급사 상품 신청·수정·취소·직인 합의·해지요청·정정요청 |
| **컬리 임직원 - MD** | 1차 검토자 (REVIEW_TWO_DEPTH의 1차 / REVIEW_ONE_DEPTH의 단독) | 1차 승인·반려·검토/수정 |
| **컬리 임직원 - OM** | 2차 검토자 (REVIEW_TWO_DEPTH 전용) | 2차 최종 승인·반려 |
| **컬리 임직원 - PRM** | 프로모션 운영 담당 | 조회·합의 운영 보조 |
| **컬리 임직원 - 합의 담당** | 수동 합의 권한자 | 수동 합의(delegate)·합의서 일괄 다운로드 |
| **컬리 임직원 - 해지/정정 담당** | 해지·정정 처리 권한자 | 해지승인·거부·즉시해지·정정승인·반려 |
| **시스템(SYSTEM)** | 자동·배치 처리 주체 | 합의대기 전환·알림 발송·합의서 미생성 감지·동시성 충돌 처리 |

**API 접근 구분**
- 공급사: `/shared/v4/` (JWT UserToken)
- 컬리 내부: `/supervisor/v4/` (X-ESCM-CONTRACT-TOKEN)

## 2. 파트너사용자(공급사) 시나리오

### 2-1. [정상] 참여 가능 프로모션 조회

| 항목 | 내용 |
|------|------|
| 진입 경로 | 파트너포털 > 프로모션 > 참여 가능 목록 |
| 사용자 행동 | 프로모션 목록·상세 조회 |
| 시스템 처리 | 본인 공급사가 참여 대상(ALL_SUPPLIERS_1P 또는 SPECIFIC_SUPPLIERS_1P 포함)인 항목만 노출 |
| 자격 검증 | 1) 파트너프로모션 존재 / 2) 미취소 / 3) 혜택 유형 일치 / 4) 신청 마감일 미초과 |
| 정책 근거 | §8 |

### 2-2. [정상] 판매가 할인 상품 지원

| 항목 | 내용 |
|------|------|
| 진입 경로 | 프로모션 상세 > [상품 지원] |
| 사용자 행동 | 1. 상품·판매가·할인금액·할인율·최대수량·분담율 입력<br>2. [검증]<br>3. [지원] |
| 시스템 처리 | 검증 통과 시 [신규제안(PROPOSAL)] 생성 |
| 정책 근거 | §5-1, §5-2, §5-3 |

### 2-3. [정상] 쿠폰 상품 지원

| 항목 | 내용 |
|------|------|
| 사용자 행동 | 최소 구매 금액·최소 구매 수량 입력 후 지원 |
| 시스템 처리 | 검증 통과 시 신규제안 생성, **금액 비교 검증 스킵** (§6-4) |
| 정책 근거 | §5, §6-4 |

### 2-4. [정상] 입고가 할인 상품 지원

| 항목 | 내용 |
|------|------|
| 사용자 행동 | 공급가 할인금액·공급가 할인율 추가 입력 |
| 검증 | 공급가 할인금액 ≤ 공급가 / 공급가 할인금액 ≤ 판매가 할인금액 |
| 정책 근거 | §5-4 |

### 2-5. [정상] 증정 상품 지원

| 항목 | 내용 |
|------|------|
| 사용자 행동 | 증정 조건 기준값·제공 유형(EACH/EVERY)·기준 유형(PRICE/QUANTITY) 입력 |
| 정책 근거 | §5-5 |

### 2-6. [정상] 다다익선(번들할인) 신청

| 항목 | 내용 |
|------|------|
| 사용자 행동 | 1. 할인종류 '다다익선' 선택<br>2. 조건수량 입력 (2 이상 양의정수)<br>3. 할인금액 입력 |
| 시스템 처리 | 동일 마스터코드에 일반 1건 + 다다익선 최대 5건까지 등록 |
| 미입력 처리 | 자동 '2' 기입 |
| 정책 근거 | §15 |

### 2-7. [정상] 상품 수정 (PROPOSAL/IN_PROGRESS만)

| 항목 | 내용 |
|------|------|
| 가능 상태 | 신규제안(PROPOSAL) / 검토중(IN_PROGRESS) |
| 사용자 행동 | 상품 정보 수정 후 저장 |
| 시스템 처리 | 상태에 따라 [신규제안] 롤백 |
| 제약 | 다다익선 신청 완료 후 **할인종류·조건수량 변경 불가** (취소 후 재신청만) |

## 3. 컬리 임직원 시나리오 - MD (1차 검토)

### 3-1. [정상] 1차 승인

| 항목 | 내용 |
|------|------|
| 진입 경로 | Admin > 프로모션 > 검토 대기 목록 |
| 사용자 행동 | 1. 신청 상품 검토<br>2. [승인] |
| 시스템 처리 | 상품 상태 [MD 승인대기] → [MD 승인완료]<br>REVIEW_ONE_DEPTH: 상품 [승인완료(APPROVED)] 진입<br>REVIEW_TWO_DEPTH: [OM 승인대기] 진입 |
| 정책 근거 | §3-2, §3-5 |

### 3-2. [정상] 1차 반려

| 항목 | 내용 |
|------|------|
| 사용자 행동 | 반려 사유 입력 후 [반려] |
| 시스템 처리 | 상품 상태 [MD 반려] → [반려(REJECTED)] |
| 가능 대상 | 신규제안·검토중 |
| 정책 근거 | §3-2, §3-3 |

### 3-3. [정상] 검토/수정 (examine)

| 항목 | 내용 |
|------|------|
| 사용자 행동 | 신청 정보 일부 수정 후 저장 (다건) |
| 시스템 처리 | 검토중(IN_PROGRESS) 진입, 수정 내역 반영 |
| 정책 근거 | §11-3 |

### 3-4. [정상] 합의요청 (approve, 복수 + SMS)

| 항목 | 내용 |
|------|------|
| 사용자 행동 | 승인된 상품들 [합의요청] |
| 시스템 처리 | 공급사에 SMS·알림톡 발송 (휴대폰번호 존재 + 미삭제 시) |
| 정책 근거 | §10-3, §11-3 |

### 3-5. [예외] 권한 없는 검토 시도

| 케이스 | 시스템 처리 |
|-------|-----------|
| MD가 2차 검토(OM 권한) 시도 | 차단 |
| MD가 본인 공급사 신청서 수정 시도 | 차단 |

## 8. 사용자 시나리오 - 예외 케이스 모음

### 8-1. 권한 없는 메뉴·API 접근

| 케이스 | 시스템 처리 |
|-------|-----------|
| 공급사가 타 공급사 신청서 조회 | 본인 데이터만 응답 |
| 공급사가 검토(승인/반려) API 호출 | 차단 |

| PRM이 검토 API 호출 | 차단 |

### 8-2. 상태별 차단

| 시도 | 차단 |
|------|------|
| 합의대기 외 상태에서 합의 | INVALID_APPLICATION_STATUS |
| 프로모션 버전 변경 후 합의 | PROMOTION_HAS_CHANGED |
| AGREED 외 상태에서 정정요청 | 차단 |
| 해지요청 외 상태에서 해지승인·거부 | 차단 |
| 이미 승인·반려된 정정 취소 | 차단 |
| PROPOSAL/IN_PROGRESS 외에서 상품 수정 | 차단 |

### 8-3. 입력값 검증 종합

| 영역 | 차단 케이스 |
|------|-----------|
| 가격 | 판매가 ≤ 0, 공급가 < 0, 할인금액 < 0, 분담율 정수 0~100 외 |
| 수량 | 최대수량 1~999,999 외, 최소 구매 금액·수량 ≤ 0 |
| 다다익선 | 조건수량 < 2, 텍스트/소수점/음수, 동일 마스터+동일 조건수량 중복 |
| 입고가 | 공급가 할인금액 > 공급가, 공급가 할인금액 > 판매가 할인금액 |
| 증정 | 기준값 < 0, 제공유형/기준유형 미선택 |
| 계산 일치 | 할인금액·할인율·할인 판매가·최대예상비용·공급가 할인율 (소수점 버림 일치) |

### 8-4. 할인금액 교차 검증 정리

| 진행중 \ 신규 | Group A | Group B | Group C |
|------------|---------|---------|---------|
| **Group A** | = 동일만 | > 초과만 | > 초과만 |
| **Group B** | < 미만만 | = 동일만 | 무관 |
| **Group C** | < 미만만 | 무관 | = 동일만 |

> 다다익선(조건수량 다름)일 때 일부 경계는 ⚠️ SOFT WARNING (팝업 후 진행)

### 8-5. 알림 발송 미보장 케이스

| 케이스 | 결과 |
|-------|------|
| 공급사 담당자 부재 | 미발송 + 경고 로그 |
| 담당자 휴대폰번호 없음 또는 무효 | 미발송 + 경고 로그 |
| SMS/알림톡 - 합의요청 | 휴대폰번호 존재 + 미삭제만 발송 |
| SMS/알림톡 - 정정 승인·반려 | 필터 없음 (담당자 전체 발송) |

## 11. 프로모션 검토 ASCII 흐름도

```
[공급사: 상품 지원]
    │
    ▼
[신규제안(PROPOSAL)] ──[취소]──→ [취소(CANCELLED)]
    │
    ▼ MD 검토
[검토중(IN_PROGRESS)] ──[MD 반려]──→ [반려(REJECTED)]
    │
    ▼ MD 승인
    ┌──────────────────────────┐
    │ REVIEW_ONE_DEPTH         │
    │   → [승인완료(APPROVED)] │
    │ REVIEW_TWO_DEPTH         │
    │   → OM 검토 → 승인/반려  │
    └──────────────────────────┘
    │
    ▼ 신청서 내 모든 상품 처리 완료 + 승인/합의 1개 이상
[합의대기(READY_TO_AGREE)]  (배치)
    │
    ├─ 공급사 직인 합의 → [합의완료(AGREED)]
    └─ 컬리 수동 합의(delegate) → [합의완료(AGREED)]
                            │
                            ├─ 정정 요청 → 승인/반려 (SMS)
                            └─ 해지 요청(직인) → 승인/거부/즉시해지 → [TERMINATED]
```

## 0. 사용자 유형 정의

| 유형 | 정의 | 주요 권한 |
|------|------|----------|
| **파트너사용자(공급사)** | 컬리에 상품을 공급하는 거래처의 담당자. 파트너포털 공급사 권한으로 로그인 | 자사 발주서 조회·확정·변경, 거래명세서 조회 |
| **컬리 임직원 - MD/AMD** | 상품 기획 담당. 발주계획 등록 및 MD요청 발주 승인 | 발주계획 등록·승인, 신상품/MD요청 과발주 승인 |
| **컬리 임직원 - SCM** | 공급망 관리 담당. 발주 검토·승인·과발주 처리 | 발주계획 등록·승인, 과발주 승인 전반 |
| **컬리 임직원 - PRM/IM/OB/FA/LC/CO/RM/CC** | 발주 내역·거래명세서 조회 권한 | 조회 전용(역할별 차이) |
| **컬리 임직원 - MASTER** | 전체 메뉴 접근 권한자 | 전 기능 |
| **시스템(SYSTEM)** | 자동·배치 처리 주체 | 자동 취소, 알림 발송, 평균 수량 갱신 등 |

> V2 발주관리 일부 메뉴는 권한 코드로 접근이 제한됨 (§5-6 참조)

## 2. 파트너사용자(공급사) 시나리오

### 2-1. [정상] 발주 알림 수신 → 발주확정 (변경 없음)

| 항목 | 내용 |
|------|------|
| 진입 경로 | 카카오 알림톡(po_created01) 수신 → 파트너포털 로그인 → 발주관리 V2 > 발주내역(입고일) |
| 사용자 행동 | 1. 발주서 상세 조회<br>2. 유통기한·수량 확인<br>3. [변경 없음] 입력 |
| 시스템 처리 | 추천일자 이후·변경 없음일 경우 즉시 [발주확정] 전이 |
| 기대 결과 | 발주확정 알림톡 발송 / 거래명세서 자동 생성 |
| 정책 근거 | §1 상태 전이, §7-3 흐름 |

### 2-2. [정상] 발주 정보 변경 → 컬리 승인대기

| 항목 | 내용 |
|------|------|
| 사용자 행동 | 1. 발주서 상세 진입<br>2. 발주수량 [변경] 버튼으로 감량 입력<br>3. 저장 |
| 시스템 처리 | 컬리 승인대기로 전이, 컬리 담당자에게 검토 요청 |
| 제약 | 수량은 **감량만 가능**, 증량은 발주예정/일CAPA 확인필요 상태에서만 허용 |
| 오류 메시지 | "발주 수량은 감량만 가능합니다. (기존수량: {n}, 수정수량: {n})" |
| 정책 근거 | §5-5, §7-6 수량 제약 |

### 2-3. [정상] 소비기한·제조일자 입력 분기

| 입력 값 | 시스템 분기 |
|---------|------------|
| 1차 비교 기준일자 이상 | 발주확정 |
| (입고예정일+판매마감일수) ~ 추천일자 미만 | 컬리 승인대기 |
| (입고예정일+판매마감일수) 미만 | 발주확정불가 (저장 차단) |

> 축산물 이력번호 입력 마감: 입고예정일 D-1 23:59:59
> 일반 정보 변경 마감: 입고예정일 D-1 17:00 (이후 저장 버튼 미노출)

### 2-4. [정상] 출고방법 변경

| 항목 | 내용 |
|------|------|
| 사용자 행동 | 발주서 출고방법 [택배 ↔ 비택배] 변경 |
| 시스템 처리 | CAPA 잔여 충분 시 변경 가능, 부족 시 차단 |
| 제약 | **입고대행 발주는 '택배'로 변경 불가** |
| 정책 근거 | §2, §5-5 |

### 2-5. [정상] 거래명세서 조회·다운로드

| 항목 | 내용 |
|------|------|
| 진입 경로 | 거래명세서 내역 메뉴 |
| 사용자 행동 | 1. 자사 거래명세서 조회<br>2. 서명된 PDF 다운로드 |
| 시스템 처리 | 거래명세서 코드(K[YYMMDD]_[7자리]) 기준 조회 |
| 권한 분리 | 자사 발주분만 조회 가능 (Supplier 권한) |
| 정책 근거 | §7-2 |

### 2-6. [예외] 발주확정 후 변경 마감 시간 경과

| 항목 | 내용 |
|------|------|
| 사용자 행동 | 입고예정일 D-1 17시 이후 발주서 진입 |
| 시스템 처리 | 저장 버튼 미노출, 변경 불가 |
| 정책 근거 | §1, §2 발주확정 후 변경 마감 |

### 2-7. [예외] 파렛트 수 미입력 (밀크런픽업)

| 항목 | 내용 |
|------|------|

## 3. 컬리 임직원 시나리오 - 발주관리 V1

### 3-1. [정상] 일반 발주 등록 (MD/SCM)

| 항목 | 내용 |
|------|------|
| 진입 경로 | 발주관리 V1 > 발주그룹 등록 |
| 사용자 행동 | 1. 상품·수량 입력<br>2. 발주그룹 생성 |
| 시스템 처리 | 과발주 검사 → 통과 시 [발주대기] → 공급사 알림 발송 |
| 정책 근거 | §4 |

### 3-2. [정상] 판매관리 화면 → [발주그룹 생성] 바로 발주

| 항목 | 내용 |
|------|------|
| 진입 경로 | 판매관리 > 발주요청 목록 |
| 사용자 행동 | 1. 발주요청 최대수량 확인<br>2. [발주그룹 생성] 버튼 |
| 시스템 처리 | 자동 발주그룹 생성, 데이터 갱신 주기 15분 |
| 정책 근거 | §3 |

### 3-3. [예외] 과발주 검토 팝업

| 발생 조건 | 시스템 처리 |
|----------|------------|
| 최근 2주 평균 발주량 대비 300% 이상 | 과발주 팝업 노출 |
| 입고예정수량 포함 판매예상일수 50일 이상 | 과발주 팝업 노출 |

> 2주 평균 발주수량 배치: 새벽 00:40, 공급상태 '발주가능/견적서대기' 대상

### 3-4. [예외] 과발주 승인 요청 (SCM 또는 SCM&MD)

| 발주구분 | 승인자 | 추가 입력 |
|---------|------|----------|
| 일반발주 | SCM | - |
| 행사(일특/라방) | SCM & MD | 승인자 공유사항 필수 |
| 행사(기획/특가) | SCM | 6개월 할인기간 일평균 판매수량 초과 |
| 신상품 | MD | 1BOX 또는 10EA 초과 |
| MD요청 | SCM (누적 초과 시 SCM & MD) | 품의서 코드 필수 |
| 긴급발주 | SCM | 누적 공급가 20M(예외 카테고리 없음) |

> 과발주 승인 생략 공급사 11개사: 넥스트키친·농협용인지점·CJ제일제당·풀무원식품·록야·매일유업·하림·케이디글로벌·엘지생활건강·팜스코

### 3-5. [정상] 발주 취소

| 취소 유형 | 트리거 | 가능 상태 |
|----------|------|----------|

| 재발주 | [재발주] 버튼 | 발주생성·공급사승인대기 |
| 기타 | [발주취소] 버튼 | 발주생성·공급사승인대기 |
| 자동 | 입고예정일 경과 미확정 → 다음날 00시 배치 | 발주생성·공급사승인대기 |

## 6. 사용자 시나리오 - 예외 케이스 모음

### 6-1. 권한 없는 메뉴 접근

| 케이스 | 시스템 처리 |
|-------|-----------|
| PRM 권한이 발주계획 등록 시도 | 메뉴 미노출 / URL 직접 접근 시 차단 |
| FBK·ITO 권한이 발주 내역(등록일) 조회 | 메뉴 미노출 |
| 공급사 권한이 타 공급사 거래명세서 조회 | 자사 데이터만 응답 |

### 6-2. 차단 처리(예외 케이스)

| 시도 | 차단 메시지 |
|------|----------|
| 발주확정·명세서생성완료 상태 취소 | "취소할 수 없는 발주 상품이 포함되어 있습니다." |
| 발주예정/일CAPA 확인필요 외 상태에서 수량 증량 | "발주 수량은 감량만 가능합니다." |
| 컬리 일괄 승인 시 다른 상태 혼재 | "컬리 승인대기 외 상품이 존재합니다." |
| 공급사 일괄 확정 시 다른 상태 혼재 | "공급사 승인대기 외 상품이 존재합니다." |
| 다른 공급사 상품 포함 확정 시도 | "%s 공급사의 상품만 확정할 수 있습니다." |
| 수정 불가 상태 상품 수정 시도 | "수정할 수 없는 상품이 포함되어 있습니다." |
| 존재하지 않는 발주 조회 | "발주 상품을 찾을 수 없습니다." |
| 입고대행 발주의 출고방법 → '택배' 변경 | 변경 차단 |

### 6-3. 동시성·정합성 예외

| 케이스 | 시스템 처리 |
|-------|-----------|
| 두 사용자가 동일 발주 동시 수정 | 낙관적 락 충돌 → "업데이트된 정보가 있어 저장 불가. 페이지 새로고침 후 재시도" |
| RMS 더블 부킹 감지 | 예약키 원복, 후보키 덮어쓰기 |
| RMS 서버 오류 | 발주계획상품 상태 변경 없이 유지 |
| 발주수량 감량 시 RMS CAPA 수정 | **감소 방향만 허용**, 증가 시 차단 |

### 6-4. V1↔V2 데이터 격리

- V1과 V2는 **상호 조회 불가** (BigQuery 테이블 분리: purchase_order_* vs neo_purchase_order_*)
- 동일 사용자가 두 화면에서 동일 발주를 찾으려 할 경우 누락 발생 가능 → §7 정책 확인 필요 항목
